# ARCHITECTURE

本文档用于说明 `OpenClaw-TZ` 的整体结构与运行思路。

## 1. 核心思路

`OpenClaw-TZ` 并不是单一应用，而是一个围绕 OpenClaw 代理运行时构建的**工作区型仓库**。

它由四类核心内容组成：

1. **代理人格与行为约束**
2. **用户上下文与长期记忆**
3. **技能与自动化能力**
4. **实际业务材料与附属项目**

## 2. 架构分层

### A. 交互层

负责接收外部消息与回传结果。

典型入口包括：

- Telegram
- OpenClaw 会话
- 定时任务/心跳

### B. 代理层

由 `taizi` 代理承担直接交互职责，并可按组织结构与其他代理协作。

例如：

- `taizi`
- `zhongshu`
- `shangshu`
- 其他职能代理

### C. 工作区层

本仓库即工作区层，主要承载：

- `SOUL.md`：人设
- `AGENTS.md`：工作协议
- `USER.md`：用户信息
- `IDENTITY.md`：身份设定
- `memory/`：长期记忆
- `skills/`：扩展技能

### D. 业务资产层

用于保存与具体工作相关的材料，例如：

- `reports/`
- `hr2/`
- `edict/`
- 截图、文档、演示材料

## 3. 简化数据流

```text
用户消息
   ↓
OpenClaw 渠道接入（如 Telegram）
   ↓
taizi 代理
   ↓
读取人格/协议/记忆/技能
   ↓
执行查询、写作、自动化、协作
   ↓
返回结果给用户
```

若涉及多代理协作，可进一步变为：

```text
用户 → taizi → zhongshu → shangshu / 其他职能代理 → taizi → 用户
```

## 4. 关键文件角色

| 文件/目录 | 作用 |
|---|---|
| `AGENTS.md` | 工作协议与输出格式约束 |
| `SOUL.md` | 角色设定与语气风格 |
| `IDENTITY.md` | 自我身份定义 |
| `USER.md` | 用户背景与偏好 |
| `HEARTBEAT.md` | 周期关注任务 |
| `memory/` | 记忆与长期记录 |
| `skills/` | 可加载技能 |
| `reports/` | 报告与脚本 |
| `hr2/` | 业务项目/样例项目 |
| `edict/` | 独立项目（当前为嵌套仓库） |

## 5. 运行依赖关系

该仓库依赖以下外部条件：

- OpenClaw 主程序
- 正确配置的 `~/.openclaw/openclaw.json`
- 可用模型服务
- 渠道配置（如 Telegram）
- 本地命令执行环境

若缺失其中任一项，仓库本身仍可作为文件集合存在，但代理能力会不完整。

## 6. 当前架构特点

### 优点

- 适合长期演化
- 人设、协议、记忆分层清晰
- 易于扩展技能
- 适合与多代理协同模型结合

### 当前不足

- 仓库中混有实验性与业务性目录
- 部分目录尚未完全工程化清理
- `edict/` 是嵌套仓库，边界仍需整理
- 部分材料更像工作现场，而非标准模板

## 7. 建议演进方向

如果后续要把它做成更成熟的公开项目，建议继续拆分：

### 方案一：保持工作区仓库定位

保留当前结构，但强化：

- 文档
- 安全边界
- 忽略规则
- 初始化步骤

### 方案二：拆为模板仓库 + 私有工作区

- 公共仓库：只保留模板、技能、示例配置、文档
- 私有工作区：保留真实记忆、脚本、业务材料、凭证相关配置
